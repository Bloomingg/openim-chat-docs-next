本页只处理当前登录用户的 `UserInfo`。其他用户的公开资料见[获取指定用户资料](/sdk/flutter/user/retrieving-users/retrieve-users)。管理员修改用户资料时，应由可信后端完成权限校验和审计。

由管理员维护组织通讯录、同步应用用户目录或修改其他账号资料时，应在可信后端调用服务端能力；管理员 Token 不能进入 Flutter 客户端。

## 读取当前用户资料

```dart
Future<UserInfo> loadCurrentUser() async {
  final user = await OpenIM.iMManager.userManager.getSelfUserInfo();
  setCurrentUser(user);
  return user;
}
```

Future 成功后返回当前账号 `UserInfo`，可用于初始化头像、昵称和设置页。该查询不会触发资料更新 callback。

## 更新当前用户资料

`setSelfInfo()` 接收可选命名参数。只传发生变化的字段，避免资料表单意外覆盖消息接收设置。

```dart
Future<void> updateMyProfile({
  String? nickname,
  String? faceURL,
  String? ex,
}) async {
  await OpenIM.iMManager.userManager.setSelfInfo(
    nickname: nickname?.trim(),
    faceURL: faceURL?.trim(),
    ex: ex,
  );

  await loadCurrentUser();
}
```

Future 成功表示资料设置请求完成。重新查询和 `onSelfInfoUpdated` callback 是独立阶段；重新查询失败不应被误报为写入失败。

资料编辑表单只应提交 `nickname`、`faceURL`、`ex` 等用户实际修改的字段。不要把 `userID`、创建时间或从旧 `UserInfo` 快照复制出的其他字段一起传回；消息通知设置也应单独构造调用，避免保存昵称时覆盖 `globalRecvMsgOpt`。

生产代码应分别处理写入与重新查询错误：写入失败时保留编辑状态；写入成功但重新查询失败时，保留提交成功状态并允许用户稍后刷新，而不是把它显示成资料保存失败。

### 更新扩展字段

`ex` 是完整字符串，会整体覆盖旧值。多个业务模块共用时，应先读取当前值、解析并合并各自命名空间，再写回完整 JSON。多端并发更新需要由业务后端提供版本控制。

```dart
import 'dart:convert';

Map<String, dynamic> parseProfileExtra(String? ex) {
  if (ex == null || ex.isEmpty) return {};

  try {
    return Map<String, dynamic>.from(jsonDecode(ex) as Map);
  } catch (_) {
    return {};
  }
}

Future<void> updateProfileExtra(Map<String, dynamic> patch) async {
  final current = await OpenIM.iMManager.userManager.getSelfUserInfo();
  final merged = {...parseProfileExtra(current.ex), ...patch};

  await OpenIM.iMManager.userManager.setSelfInfo(
    ex: jsonEncode(merged),
  );
}
```

`ex` 只适合轻量、低敏感度且与聊天展示直接相关的数据。组织、权限、风控、审计、搜索索引和后台管理字段应由可信后端保存。

### 设置全局消息接收选项

```dart
await OpenIM.iMManager.userManager.setSelfInfo(
  globalRecvMsgOpt: ReceiveMessageOpt.notNotify,
);
```

`ReceiveMessageOpt.receive` 表示正常接收，`notReceive` 表示不接收，`notNotify` 表示接收但不通知。不要调用 `ConversationManager.setGlobalRecvMessageOpt()`；固定版本中的该方法不可用。

## 调用后的状态变化

- `getSelfUserInfo()` 成功后，以返回的 `UserInfo` 建立当前账号资料快照；该查询不会触发更新 callback。
- `setSelfInfo()` 的 Future 成功表示本次设置请求完成，不等同于 callback 已到达。
- `onSelfInfoUpdated` 携带当前账号最终的 `UserInfo`，资料页、用户菜单和全局账号状态应使用 callback 数据覆盖旧快照。
- 页面需要立即确认服务端最终值时，可以在写入成功后重新调用 `getSelfUserInfo()`；重新查询失败不应被误报为写入失败。

当前账号只有一份 `UserInfo`。退出登录或切换账号时必须清理旧账号快照，不能把上一账号的 callback 数据写入当前状态。

## 监听资料变化

本页归属 `onSelfInfoUpdated` callback。应用应把它与在线状态 callback 组合进同一个应用级 `OnUserListener`，并只设置一次。

```dart
void handleSelfInfoUpdated(UserInfo user) {
  setCurrentUser(user);
}
```

组合 listener 与唯一设置入口见[事件概览](/sdk/flutter/events/overview-events)。退出或切换账号时，应让应用状态层停止接收旧账号写入，并在新账号登录后以查询结果重新建立快照。
