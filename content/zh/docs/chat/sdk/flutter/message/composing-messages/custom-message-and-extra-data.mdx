OpenIMSDK 的消息扩展数据需要先确定作用范围：卡片、邀请、订单等完整业务对象使用自定义消息；需要随消息同步的轻量字段可以在发送前写入 `Message.ex`；只影响当前设备界面的状态写入 `localEx`。SDK 不提供在消息发送后任意追加共享键值对的通用接口。

## 选择数据落点

| 场景                               | 推荐方式                       | 是否同步到其他客户端 |
| ---------------------------------- | ------------------------------ | -------------------- |
| 订单卡片、任务卡片等结构化业务内容 | `createCustomMessage()` 后发送 | 是                   |
| 消息来源、展示版本等轻量字段       | 发送前设置 `Message.ex`        | 是                   |
| 折叠、选中、本地处理标记           | `setMessageLocalEx()`          | 否                   |

## 发送带共享数据的消息

`data`、`extension` 和 `description` 都是必填字符串。建议在 `data` 中保存带版本的 JSON，用 `extension` 标识业务类型，并把 `description` 作为解析失败时的降级文案。

### 参数说明

| 参数 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `data` | `String` | 必填 | 序列化后的业务数据。 |
| `extension` | `String` | 必填 | 业务消息类型或协议标识，供接收端选择解析和渲染逻辑。 |
| `description` | `String` | 必填 | 无法解析业务数据时的降级展示文案。 |

```dart
final message = await OpenIM.iMManager.messageManager.createCustomMessage(
  data: jsonEncode({
    'version': 1,
    'type': 'order-card',
    'orderID': orderID,
  }),
  extension: jsonEncode({'schema': 'com.example.order'}),
  description: '订单卡片',
);

final sentMessage = await OpenIM.iMManager.messageManager.sendMessage(
  message: message,
  userID: receiverUserID,
  offlinePushInfo: OfflinePushInfo(title: '订单更新', desc: '查看订单详情'),
);
```

创建成功只得到待发送的 `Message`。发送成功后，发送端先检查 `sentMessage.clientMsgID` 不为 null 且非空，再用它合并列表；接收端按[接收消息](/sdk/flutter/message/receiving-messages/receive-messages)处理新消息。先检查 `message.contentType == MessageType.custom`，再读取 `message.customElem` 并用 `try/catch` 解析 JSON。未知版本或字段缺失应降级为通用提示，不要让消息列表渲染中断。

不要把需要参与权限判断、支付状态或审批流的最终状态只放在自定义消息中。这类状态应由业务服务校验和持久化，消息只承载展示入口或状态快照。

## 写入共享扩展字段

`Message.ex` 是随消息发送和存储的扩展字符串。接收端通过新消息 listener 获取消息，或之后查询历史消息时，都可以从同一条 `Message` 读取 `ex`。它适合保存消息来源、展示版本或其他轻量且不敏感的附加信息。

创建消息后、调用 `sendMessage()` 前，为 `ex` 写入完整字符串：

```dart
final message = await OpenIM.iMManager.messageManager.createTextMessage(
  text: '请查看最新方案',
);

message.ex = jsonEncode({
  'source': 'project-board',
  'schemaVersion': 1,
});

final sentMessage = await OpenIM.iMManager.messageManager.sendMessage(
  message: message,
  userID: receiverUserID,
);
```

`Message.ex` 没有自动合并对象的语义。多个业务模块共用该字段时，应约定字段命名和版本，并在写入前生成完整值。不要把访问令牌、密钥、隐私信息或需要服务端持续校验的权威状态放入 `ex`。

## 写入本地附加数据

`setMessageLocalEx()` 只修改当前设备上的 `localEx`，适合保存 UI 展开状态或本地处理标记，不会同步给其他端，也不应被当作服务端业务状态。调用前应由当前消息页面保留 `conversationID`，并检查消息的 nullable `clientMsgID`。

### 参数说明

| 参数 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `conversationID` | `String` | 必填 | 目标消息所在的会话 ID。 |
| `clientMsgID` | `String` | 必填 | 要更新的消息 ID；从目标 `Message.clientMsgID` 取得并先检查非空。 |
| `localEx` | `String` | 必填 | 本地扩展的完整值；不会自动合并旧 JSON。 |

```dart
final clientMsgID = message.clientMsgID;
if (clientMsgID == null || clientMsgID.isEmpty) {
  throw StateError('Message has no clientMsgID');
}

await OpenIM.iMManager.messageManager.setMessageLocalEx(
  conversationID: conversationID,
  clientMsgID: clientMsgID,
  localEx: jsonEncode({'expanded': true}),
);
```

`setMessageLocalEx()` 写入完整字符串，不会自动合并旧值；需要保留字段时，先解析 `message.localEx`，在应用层合并后再写回。Future 成功分别表示消息对象创建、发送请求或本地扩展写入完成。创建和本地写入不会触发新消息事件；远端接收仍按[接收消息](/sdk/flutter/message/receiving-messages/receive-messages)处理。

## 事件和校验

普通聊天自定义消息仍由 `OnAdvancedMsgListener.onRecvNewMessage` 或 `onRecvOfflineNewMessage` 进入消息列表。服务端单独下发的自定义业务通知则由 `OnCustomBusinessListener.onRecvCustomBusinessMessage` 接收；它的载荷是业务字符串，不是 `Message`。

`OnCustomBusinessListener` 是 singleton listener。应用应只在[事件概览](/sdk/flutter/events/overview-events)的集中入口调用一次 `setCustomBusinessListener()`，并把回调交给本页的处理函数：

```dart
void handleCustomBusinessMessage(String payload) {
  try {
    final decoded = jsonDecode(payload);
    if (decoded is! Map<String, dynamic>) {
      throw const FormatException('Expected a JSON object');
    }

    final eventID = decoded['eventID'];
    final eventType = decoded['type'];
    if (eventID is! String || eventID.isEmpty || eventType is! String) {
      throw const FormatException('Missing eventID or type');
    }

    applyCustomBusinessEvent(
      eventID: eventID,
      eventType: eventType,
      payload: decoded,
    );
  } on FormatException catch (error) {
    reportInvalidCustomBusinessPayload(error, payload);
  }
}
```

应用应按自己的协议验证字段，并使用业务 ID 或幂等键去重和合并状态；不要依赖事件到达顺序或数组位置。listener 设置成功只表示 native 侧已注册回调，不代表任何通知已经到达。需要恢复权威状态时，仍应重新查询对应业务服务。

创建消息对象只会在 Future 中返回本地待发送的 `Message`，不会触发接收事件。`sendMessage()` 成功后直接使用返回的 `Message` 合并发送方列表；其他客户端再通过消息 listener 获得消息。`setMessageLocalEx()` 只更新本地数据，不会产生共享消息事件。

验证附加数据时，重点确认这些结果：

- `createCustomMessage()`、`sendMessage()` 或 `setMessageLocalEx()` 成功时返回预期数据，失败时进入 `catch`。
- 发送前写入 `ex` 的消息，可以在其他客户端收到的 `Message` 和之后查询到的历史消息中读取相同扩展值。
- 需要同步给其他成员的数据，可以在另一个已登录客户端通过消息 listener 收到。
- 只写入 `localEx` 的状态不会出现在其他用户或其他设备的共享消息内容中。
- 失败日志包含 `code`、`message` 和必要的目标 ID，便于和服务端日志对齐。
