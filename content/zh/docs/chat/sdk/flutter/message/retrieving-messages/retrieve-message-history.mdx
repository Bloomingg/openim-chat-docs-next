历史消息以 `conversationID` 为范围。第一页不传 `startMsg`；继续向更早记录分页时，把当前列表最早一条 `Message` 作为下一页游标。

`conversationID` 可以来自会话列表或会话事件。如果页面只有用户 ID 或群组 ID，先调用 `getConversationIDBySessionType(sourceID:, sessionType:)` 换取，不要自行拼接会话 ID。

OpenIMSDK 的历史读取以会话为中心，单聊和群聊共用同一套分页模型。进入会话时先请求第一页；用户上拉或滚动到历史边界时，再传入当前最早消息。

## 按会话读取消息

```dart
Future<AdvancedMessage> loadHistoryPage({
  required String conversationID,
  Message? startMessage,
}) {
  return OpenIM.iMManager.messageManager.getAdvancedHistoryMessageList(
    conversationID: conversationID,
    startMsg: startMessage,
    count: 30,
    viewType: GetHistoryViewType.history,
  );
}

final page = await loadHistoryPage(
  conversationID: conversationID,
  startMessage: oldestMessage,
);
prependMessages(page.messageList ?? const <Message>[]);
```

调用方可以用第一页初始化消息列表，将后续页插入列表较早的一端。合并时应先确认 `clientMsgID` 非空，再结合当前 `conversationID` 去重，避免登录同步、新消息 listener 和历史分页返回同一条消息。

```dart
Message? historyCursor;
var reachedHistoryEnd = false;

Future<void> loadOlderMessages(String conversationID) async {
  if (reachedHistoryEnd) return;

  final page = await loadHistoryPage(
    conversationID: conversationID,
    startMessage: historyCursor,
  );
  final messages = page.messageList ?? const <Message>[];

  prependMessages(messages);
  if (messages.isNotEmpty) historyCursor = messages.last;
  reachedHistoryEnd = page.isEnd == true || messages.isEmpty;
}
```

### 参数说明

| 参数 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `conversationID` | `String?` | 业务必填 | 要读取的会话 ID。 |
| `startMsg` | `Message?` | 可选 | 分页边界；第一页为 `null`，后续传当前最早消息。 |
| `viewType` | `GetHistoryViewType` | 可选 | 普通聊天记录使用 `GetHistoryViewType.history`。 |
| `count` | `int?` | 可选 | 本页数量，未传时 SDK 默认 40。 |

### 返回结果

返回 `AdvancedMessage.messageList`、`isEnd`、`lastMinSeq`，以及可能的 `errCode`、`errMsg`。空列表且 `isEnd == true` 时停止分页；每条消息先检查 nullable `clientMsgID` 再去重，不要使用数组下标或当前列表长度作为游标。空列表但同步尚未完成时保留原游标，待同步完成后重试当前页。

历史查询的 Future 成功后，返回值就是本次请求的消息快照，不需要等待 listener 才能渲染。读取历史本身不会触发新消息 callback。

## 群聊消息列表

如果业务入口只有 `groupID`，先取得群组对应的 `conversationID`，再使用同一个历史接口：

```dart
final conversationID = await OpenIM.iMManager.conversationManager
    .getConversationIDBySessionType(
  sourceID: groupID,
  sessionType: ConversationType.superGroup,
);

final firstPage = await OpenIM.iMManager.messageManager
    .getAdvancedHistoryMessageList(
  conversationID: conversationID,
  count: 30,
  viewType: GetHistoryViewType.history,
);
```

`getConversationIDBySessionType()` 和历史查询都直接通过 Future 返回当前快照，查询本身不会触发会话或消息 listener。群聊页面打开期间，新消息应通过[接收消息](/sdk/flutter/message/receiving-messages/receive-messages)中的统一处理器合并，本页不重复设置 listener。

## 选择读取方向

`getAdvancedHistoryMessageListReverse()` 使用相同参数，从起点读取较新的方向，适合按 ID 定位后补齐后文。向前读取时下一页以当前较新边界为 `startMsg`，两种方向不要在同一滚动边界混用。查询 Future 只返回快照，不触发新消息事件；页面打开期间继续合并[接收消息](/sdk/flutter/message/receiving-messages/receive-messages)中的增量。查询快照、事件增量和重新查询校准是三个独立阶段。

固定 Flutter SDK 使用完整 `Message` 作为分页边界，不是 WASM 的 `clientMsgID` 字符串游标。新页面应统一一种读取方向与一种边界规则，减少重复消息、顺序反转和滚动位置跳动。

## 游标和空列表

不要用数组下标、当前分页长度或展示时间作为下一页游标。消息撤回、删除、本地同步和多端登录都可能改变当前列表；应保留 SDK 返回的边界 `Message`，并以非空 `clientMsgID` 作为列表去重标识。

需要刷新当前消息列表时，可以保留现有列表并重新读取边界页。重新查询、listener 增量和此前 Future 完成互不替代，三条路径都应按相同会话与 `clientMsgID` 合并。
