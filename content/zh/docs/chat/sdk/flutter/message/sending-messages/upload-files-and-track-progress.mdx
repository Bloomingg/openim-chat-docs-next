媒体消息通过 `sendMessage()` 进入 SDK 上传链路时，使用唯一的 `OnMsgSendProgressListener` 关联 `clientMsgID` 与进度。应在消息状态层设置一次，不要在消息列表项中重复设置。

上传百分比、失败提示和重试按钮属于应用 UI 状态。SDK listener 只提供进度与完成信号，应用需要把回调关联到本地上传任务或消息气泡。

## 注册上传处理器

```dart
OpenIM.iMManager.messageManager.setMsgSendProgressListener(
  OnMsgSendProgressListener(
    onProgress: (clientMsgID, progress) {
      updateUploadProgress(clientMsgID, progress);
    },
  ),
);
```

`progress` 用于展示当前上传进度；最终发送状态仍以 `sendMessage()` 返回的 `Message` 为准。固定版本没有公开的移除方法，切换账号时应隔离旧状态。

## 独立上传文件

需要先上传、后组装消息时，设置唯一的 `OnUploadFileListener`，再以稳定的业务 `id` 发起上传：

### 参数说明

| 参数 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `id` | `String` | 必填 | 关联上传任务与 listener 回调的稳定业务 ID。 |
| `filePath` | `String` | 必填 | 当前平台可读的文件路径。 |
| `fileName` | `String` | 必填 | 上传使用的原始文件名。 |
| `contentType` | `String?` | 可选 | 文件 MIME 类型。 |
| `cause` | `String?` | 可选 | 业务侧附加的上传原因。 |

```dart
OpenIM.iMManager.setUploadFileListener(
  OnUploadFileListener(
    onUploadProgress: (id, fileSize, streamSize, storageSize) {
      updateFileProgress(id, streamSize, fileSize);
    },
    onComplete: (id, size, url, type) {
      handleUploadedFile(id, size, url, type);
    },
  ),
);

await OpenIM.iMManager.uploadFile(
  id: uploadID,
  filePath: filePath,
  fileName: fileName,
  contentType: contentType,
);
```

`onComplete` 返回的 URL 和大小用于构造对应的 `PictureInfo`、`SoundElem`、`VideoElem` 或 `FileElem`，随后调用 `create*MessageByURL()` 与 `sendMessageNotOss()`。

## 创建并发送文件消息

文件消息仍遵循“先创建 `Message`、再调用 `sendMessage()`”的两步模型。传给消息工厂的路径必须是当前 Android 或 iOS 应用可读取的本地路径。

```dart
Future<Message> sendFileWithProgress({
  required String filePath,
  required String fileName,
  String? userID,
  String? groupID,
}) async {
  final message = await OpenIM.iMManager.messageManager.createFileMessage(
    filePath: filePath,
    fileName: fileName,
  );

  return OpenIM.iMManager.messageManager.sendMessage(
    message: message,
    userID: userID,
    groupID: groupID,
    offlinePushInfo: OfflinePushInfo(
      title: '文件消息',
      desc: fileName,
    ),
  );
}
```

单聊只传 `userID`，群聊只传 `groupID`。图片、音频和视频使用各自的消息工厂，创建后仍交给同一个 `sendMessage()`；业务已经上传并取得 URL 时，才改用 `create*MessageByURL()` 和 `sendMessageNotOss()`。

## 关联进度到消息

如果界面允许同时上传多个文件，不要按文件名或列表位置匹配进度。同名文件、重试及跨会话并发都可能造成状态串扰：SDK 媒体发送进度按 `clientMsgID` 关联，独立上传任务按业务生成且唯一的 `id` 关联。

创建消息后保存它的非空 `clientMsgID` 与本地任务 ID；`onProgress` 更新对应消息气泡，发送 Future 成功后再用服务端确认的 `Message` 替换本地待发送项。发送失败时保留文件名、目标会话和错误状态，但不要记录 Token、完整媒体 URL 或敏感文件内容。

两个 listener 都保存在 manager singleton 上，后设值会覆盖前值，固定版本没有移除方法。应在应用统一状态层各设置一次；切换账号时清空上一账号的任务映射和 UI 订阅，避免同名任务或旧回调串到新账号。

上传完成不等于消息发送成功。`uploadFile()` 的 Future、上传回调、消息发送 Future 和接收事件是独立阶段，应分别维护错误与重试状态；并发上传必须按 `id` 或消息的 `clientMsgID` 关联，不要按文件名或列表位置匹配。

重试时应重新创建消息，或在业务确认独立上传已经成功后复用资源 URL；不要把旧任务的进度写入新的消息气泡。需要恢复当前消息列表时，再通过历史查询建立快照，不能把上传完成 callback 当作消息已经发送。

## 验证上传结果

- 确认 SDK 已登录且连接正常，再开始发送媒体消息。
- 使用较大的图片或文件验证进度会从等待、上传中变为完成。
- 确认 `sendMessage()` 返回的 `Message` 已按 `clientMsgID` 合并到当前会话。
- 并发发送多个同名文件，确认进度不会互相覆盖。
- 切换页面或账号后再次发送，确认 singleton listener 没有把旧账号状态写入当前 UI。
